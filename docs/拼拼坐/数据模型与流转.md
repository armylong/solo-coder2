# 数据模型与流转

## 一、数据库表结构

### 1.1 tb_ppz_user（用户扩展表）

存储拼拼坐相关的用户扩展信息，与主用户表（tb_user）通过 uid 关联。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 主键ID |
| uid | INTEGER UNIQUE | 用户ID（关联 tb_user.uid） |
| car_count | INTEGER | 车辆数量（>0=司机，=0=普通用户） |
| status | INTEGER | 用户状态：1-正常，2-已封禁 |
| driver_status | INTEGER | 司机状态：1-正常，2-已封禁 |
| ban_reason | TEXT | 封禁原因 |
| banned_at | DATETIME | 封禁时间 |
| unbanned_at | DATETIME | 解封时间 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

**设计说明**：
- `car_count > 0` 表示该用户是司机（有审核通过的车辆）
- `car_count = 0` 表示该用户是普通用户
- `status` 和 `driver_status` 分离，支持仅封禁司机角色而不影响用户基本功能
- 采用 `GetOrCreateByUid` 模式，首次访问时自动创建记录

---

### 1.2 tb_ppz_cars（车辆表）

存储审核通过的车辆信息。

| 字段 | 类型 | 说明 |
|------|------|------|
| car_id | INTEGER PK | 车辆ID（主键） |
| uid | INTEGER | 用户ID |
| car_model | TEXT | 车辆型号 |
| car_license_photo | TEXT | 行驶证照片URL |
| driver_license_photo | TEXT | 驾驶证照片URL |
| license_plate | TEXT | 车牌号 |
| car_color | TEXT | 车辆颜色 |
| seats | INTEGER | 乘客座位数 |
| car_photo | TEXT | 车辆照片URL |
| description | TEXT | 车辆简介 |
| status | INTEGER | 状态：1-正常，0-删除 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

**设计说明**：
- 只有审核通过的车辆才会写入此表
- 删除采用软删除（status=0）
- 车辆数据来源于审核通过时的 audit_data 快照

---

### 1.3 tb_ppz_car_audit（车辆审核表）

存储车辆审核记录，是车辆管理的核心表。

| 字段 | 类型 | 说明 |
|------|------|------|
| audit_id | INTEGER PK | 审核ID（主键） |
| uid | INTEGER | 用户ID |
| car_id | INTEGER | 车辆ID（新增时为0，审核通过后回填） |
| audit_data | TEXT(JSON) | 提交的车辆数据（JSON格式） |
| audit_status | INTEGER | 审核状态：0-已删除，1-待审核，2-已通过，3-已驳回 |
| audit_reason | TEXT | 审核理由 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

**audit_data JSON 结构**：

```json
{
  "car_model": "奔驰GLC300L",
  "car_license_photo": "https://...",
  "driver_license_photo": "https://...",
  "license_plate": "京A12345",
  "car_color": "蓝色",
  "seats": 5,
  "car_photo": "https://...",
  "description": "这是一辆奔驰GLC300L..."
}
```

**设计说明**：
- 新增车辆时 `car_id = 0`，审核通过后回填
- 编辑车辆时更新 `audit_data` 并重置 `audit_status = 1`（待审核）
- 审核通过后，从 `audit_data` 中提取数据创建/更新车辆表记录
- 封禁司机时，所有待审核和已通过的审核记录都会被置为驳回

---

### 1.4 tb_ppz_car_audit_log（审核操作日志表）

记录每次审核状态变更，形成完整的操作链。

| 字段 | 类型 | 说明 |
|------|------|------|
| audit_log_id | INTEGER PK | 审核日志ID（主键） |
| audit_id | INTEGER | 审核ID |
| uid | INTEGER | 用户ID |
| car_id | INTEGER | 车辆ID |
| audit_data | TEXT(JSON) | 审核时的车辆数据快照 |
| audit_status | INTEGER | 审核状态快照 |
| audit_reason | TEXT | 审核理由 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

**设计说明**：
- 每次审核状态变更都会写入一条日志
- 日志记录的是变更后的状态快照
- 可通过 `audit_id` 查询某次审核的完整变更历史

---

### 1.5 tb_ppz_map_address（常用地址表）

存储用户的常用地址信息。

| 字段 | 类型 | 说明 |
|------|------|------|
| address_id | INTEGER PK | 地址ID（主键） |
| uid | INTEGER | 用户ID |
| remark | TEXT | 地址备注（如：我家、公司） |
| gaode_data | TEXT(JSON) | 高德地图返回的地址数据 |
| sort | INTEGER | 排序值（越大越靠前） |
| status | INTEGER | 状态：1-正常，0-删除 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

**设计说明**：
- 新建地址时 `sort` 自动取当前最大值 +1
- 删除采用软删除（status=0）
- `gaode_data` 存储高德地图逆地理编码返回的完整 JSON 数据

---

### 1.6 tb_ppz_business_area（运营区域表）

存储运营区域及其围栏数据。

| 字段 | 类型 | 说明 |
|------|------|------|
| area_id | INTEGER PK | 区域ID（自增主键） |
| area_name | TEXT | 区域名称 |
| area_fence | TEXT(JSON) | 区域围栏数据（JSON格式，3D数组） |
| status | INTEGER | 状态：0-已删除，1-正常，2-停用 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

**area_fence JSON 结构**：

```json
[
  [
    [[116.123456, 39.123456], [116.234567, 39.234567], ...],
    [[116.111111, 39.111111], [116.222222, 39.222222], ...]
  ],
  [
    [[117.123456, 40.123456], [117.234567, 40.234567], ...]
  ]
]
```

**设计说明**：
- `area_fence` 实现 Scan/Value 接口，自动序列化/反序列化 JSON
- 围栏数据为3D数组：`[polygon1, polygon2, ...]`，每个 polygon 是 `[outerRing, hole1, ...]`
- 删除采用软删除（status=0）
- 支持启用/停用切换（status=1/2）

---

### 1.7 tb_ppz_business_route（运营路线表）

存储运营区域之间的路线信息。

| 字段 | 类型 | 说明 |
|------|------|------|
| route_id | INTEGER PK | 路线ID（自增主键） |
| route_name | TEXT | 路线名称 |
| a_area_id | INTEGER | A端区域ID（关联 tb_ppz_business_area） |
| b_area_id | INTEGER | B端区域ID（关联 tb_ppz_business_area） |
| status | INTEGER | 状态：0-已删除，1-正常，2-停用 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

**设计说明**：
- 一条路线连接两个运营区域（A端和B端）
- 删除运营区域前需检查是否有关联路线（`CountByAreaId`）
- 删除采用软删除（status=0）
- 支持启用/停用切换（status=1/2）

---

### 1.8 tb_ppz_trip（行程表）

存储行程信息，一个行程对应一个司机的一次出车。

| 字段 | 类型 | 说明 |
|------|------|------|
| trip_id | INTEGER PK | 行程ID（自增主键） |
| driver_id | INTEGER | 司机uid（关联 tb_ppz_user.uid） |
| status | INTEGER | 行程状态：0-已取消，1-准备中，2-行程中，3-已完成 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

**行程状态说明**：

| 状态 | 值 | 含义 |
|------|---|------|
| 已取消 | 0 | 行程被取消 |
| 准备中 | 1 | 司机接单到出发前，攒人、确认、等时间 |
| 行程中 | 2 | 司机出发了，不管接人、路上、部分到达，都是行程中 |
| 已完成 | 3 | 所有乘客都到达目的地 |

**设计说明**：
- 行程表只管行程维度的大状态，详细状态由订单表管理
- 行程先于司机出发存在（准备中），司机出发后变为行程中
- 所有乘客都到达后才变为已完成
- 准备中不锁死，3个人也能出发，后续有合适的单子还能塞进新的行程

---

### 1.9 tb_ppz_order（订单表）

存储乘客的拼车订单信息，订单先于行程存在。

| 字段 | 类型 | 说明 |
|------|------|------|
| order_id | INTEGER PK | 订单ID（自增主键） |
| uid | INTEGER | 乘客uid（关联 tb_user.uid） |
| trip_id | INTEGER | 行程ID（0=未匹配，司机接单后回填） |
| start_remark | TEXT | 出发地备注（如：我家） |
| start_gaode_data | TEXT(JSON) | 出发地高德数据（精简9字段，同地址表） |
| dest_remark | TEXT | 目的地备注 |
| dest_gaode_data | TEXT(JSON) | 目的地高德数据 |
| depart_date | TEXT | 出发日期（如：2026-05-10） |
| depart_time | TEXT | 出发时间（如：14:00） |
| time_type | INTEGER | 时间类型：1-按时出发，2-按时到达 |
| time_flex | INTEGER | 时间弹性（分钟数，如30） |
| passenger_count | INTEGER | 乘车人数 |
| is_charter | INTEGER | 是否包车：0-否，1-是 |
| order_status | INTEGER | 订单状态：0-已取消，1-匹配中，2-已接单，3-已确认，4-匹配失败 |
| cancel_reason | TEXT | 取消原因 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

**start_gaode_data / dest_gaode_data JSON 结构**（精简9字段，同 tb_ppz_map_address）：

```json
{
  "formatted_address": "山东省泰安市新泰市泉沟镇兴泉家园",
  "lng": 117.643879,
  "lat": 35.981911,
  "province": "山东省",
  "city": "泰安市",
  "district": "新泰市",
  "township": "泉沟镇",
  "adcode": "370982",
  "aoi_name": "兴泉家园"
}
```

**订单状态说明**：

| 状态 | 值 | 含义 |
|------|---|------|
| 已取消 | 0 | 订单被取消 |
| 匹配中 | 1 | 用户下单即是匹配中，系统派单/进抢单池 |
| 已接单 | 2 | 司机接单，trip_id 回填 |
| 已确认 | 3 | 司机电话联系后确认该乘客 |
| 匹配失败 | 4 | 无人接单，出发时间已过 |

**设计说明**：
- 订单先于行程存在，用户下单时 trip_id=0，司机接单时创建行程并回填 trip_id
- 出发地/目的地内联存储 gaode_data，不关联 address_id，订单数据自包含不依赖地址表变动
- 后期通过经纬度反查匹配用户的地址 id 做展示
- `gaode_data` 实现 Scan/Value 接口，自动序列化/反序列化 JSON
- 订单状态管匹配和乘客确认，行程状态管物理进度，两层状态机独立运转

## 三、核心数据流转

### 3.1 车辆审核流转

```
                    ┌─────────────┐
                    │  用户添加车辆  │
                    └──────┬──────┘
                           │
                           ▼
              ┌────────────────────────┐
              │ tb_ppz_car_audit       │
              │ audit_status = 1       │
              │ car_id = 0             │
              │ audit_data = {车辆信息}  │
              └────────────┬───────────┘
                           │
                    ┌──────▼──────┐
                    │  管理员审核   │
                    └──────┬──────┘
                           │
                ┌──────────┼──────────┐
                │                     │
          审核通过                 审核驳回
                │                     │
                ▼                     ▼
  ┌──────────────────────┐  ┌──────────────────────┐
  │ audit_status = 2     │  │ audit_status = 3     │
  │                      │  │ audit_reason = 驳回原因│
  │ 1. 创建车辆记录       │  └──────────────────────┘
  │    tb_ppz_cars       │           │
  │ 2. 回填 car_id       │           ▼
  │ 3. 递增 car_count    │  ┌──────────────────────┐
  └──────────────────────┘  │ 用户可编辑重新提交     │
                            │ → audit_status = 1    │
                            │ → audit_data 更新     │
                            └──────────────────────┘
```

### 3.2 审核日志流转

每次审核状态变更都会同步写入日志表：

```
审核状态变更 → CreateFromAudit(audit) → tb_ppz_car_audit_log

记录内容：
- audit_id: 关联的审核记录ID
- uid: 用户ID
- car_id: 车辆ID
- audit_data: 当前车辆数据快照
- audit_status: 变更后的状态
- audit_reason: 审核理由
```

### 3.3 司机封禁流转

```
                    ┌─────────────┐
                    │  管理员封禁   │
                    └──────┬──────┘
                           │
                           ▼
              ┌────────────────────────┐
              │ tb_ppz_user            │
              │ driver_status = 2      │
              │ ban_reason = 封禁原因    │
              │ banned_at = NOW()      │
              └────────────┬───────────┘
                           │
                           ▼
              ┌────────────────────────┐
              │ 遍历该用户所有审核记录    │
              │ audit_status > 0 的     │
              │ → 置为 audit_status = 3 │
              │ (待审核/已通过 → 已驳回)  │
              └────────────┬───────────┘
                           │
                           ▼
              ┌────────────────────────┐
              │ 用户端影响：            │
              │ - 无法添加新车辆        │
              │ - 无法编辑车辆          │
              │ - checkDriver 返回 false│
              └────────────────────────┘

                    ┌─────────────┐
                    │  管理员解封   │
                    └──────┬──────┘
                           │
                           ▼
              ┌────────────────────────┐
              │ tb_ppz_user            │
              │ driver_status = 1      │
              │ unbanned_at = NOW()    │
              └────────────────────────┘
              (注意：解封不会恢复车辆审核状态，
               司机需重新提交车辆审核)
```

### 3.4 地址管理流转

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  地址列表页面  │────→│  地图选点页面  │────→│  保存地址     │
│  (address/)  │     │(address-map/)│     │              │
└──────────────┘     └──────────────┘     └──────┬───────┘
                                                 │
                                                 ▼
                                  ┌──────────────────────────┐
                                  │ address_id > 0 ?         │
                                  │   是 → UPDATE 地址        │
                                  │   否 → INSERT 新地址      │
                                  │     sort = max_sort + 1  │
                                  └──────────────────────────┘
```

## 五、数据一致性保障

### 5.1 审核通过的数据同步

审核通过时需要同步操作多张表，保证数据一致性：

1. 创建/更新 `tb_ppz_cars` 记录
2. 回填 `tb_ppz_car_audit.car_id`
3. 更新 `tb_ppz_car_audit.audit_status = 2`
4. 写入 `tb_ppz_car_audit_log` 日志
5. 递增 `tb_ppz_user.car_count`

### 5.2 封禁司机的级联操作

封禁司机时需要级联更新：

1. 更新 `tb_ppz_user.driver_status = 2`
2. 遍历该用户所有 `audit_status > 0` 的审核记录，置为驳回
3. 每条驳回的审核记录都写入日志

### 5.3 用户权限校验

所有用户端操作都会先检查司机状态：

```
checkDriverStatus(uid)
  → IsDriverBanned(uid)
    → 如果已封禁，返回错误"您的司机状态异常, 不能管理车辆"
```

## 六、数据库自动迁移

### 6.1 AutoMigrate 机制

所有数据表的 `CreateTable` 方法末尾都调用了 `sqlite.DB.AutoMigrate()`，实现结构体字段与数据库列的自动同步。

**工作原理**：
1. `CREATE TABLE IF NOT EXISTS` 创建表（仅首次生效，不会更新已有表的列）
2. `AutoMigrate` 通过 `PRAGMA table_info` 查询已有列
3. 对比结构体的 `json` tag，找出缺失的列
4. 缺失的列通过 `ALTER TABLE ADD COLUMN` 自动补上

**使用方式**：
- 给结构体新增字段后，重启服务即可自动建列
- 新列只能加到表的末尾（SQLite 限制），但结构体中可按逻辑顺序排列
- 不支持自动删列（防止误删数据），需手动操作

**示例**：
```go
type TbPpzCars struct {
    CarId     int64  `json:"car_id" db:"pk"`
    Uid       int64  `json:"uid"`
    CarModel  string `json:"car_model"`
    NewField  string `json:"new_field"`  // 新增字段，重启服务后自动建列
    // ...
}
```
