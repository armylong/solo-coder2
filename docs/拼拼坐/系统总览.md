# 拼拼坐系统 - 产品文档总览

## 一、系统简介

拼拼坐是一个**本地化拼车系统**，旨在为用户提供便捷的拼车出行服务。系统采用前后端分离架构，分为**用户端（ppz-m）**和**管理端（ppz-admin）**两个独立前端应用，后端基于 Go 语言开发，使用 SQLite 作为数据存储。

### 核心定位

- **用户端**：面向拼车用户，提供注册登录、车辆管理、常用地址管理等功能
- **管理端**：面向运营管理人员，提供司机管理、车辆审核、电子围栏配置等功能

## 三、核心业务流程

### 3.1 用户注册与登录

```
用户 → 输入账号密码 → 注册/登录 → 获取登录态(Cookie) → 进入系统
```

- 注册仅需用户名+密码，无手机号验证
- 登录后通过 Cookie 维持会话
- 所有业务接口均需登录态校验

### 3.2 司机认证流程

```
用户添加车辆 → 提交审核(待审核) → 管理员审核 → 通过/驳回
                                              ↓
                                    通过 → 创建车辆记录 → 用户成为司机
                                    驳回 → 用户可编辑重新提交
```

**关键设计**：
- 车辆信息不直接写入车辆表，而是先写入审核表
- 审核通过后，才从审核表数据创建车辆记录
- 编辑车辆本质是更新审核记录并重置为待审核状态
- 用户通过 `car_count > 0` 判断是否为认证司机

### 3.3 司机封禁流程

```
管理员封禁司机 → 司机状态置为封禁 → 所有审核记录置为驳回 → 无法添加/编辑车辆
                                              ↓
管理员解封司机 → 司机状态恢复正常 → 可重新添加车辆（需重新审核）
```

### 3.4 地址管理流程

```
用户进入地址管理 → 选择出发地/目的地 → 地图选点 → 填写备注 → 保存地址
```

- 地址分为「出发地」和「目的地」两种类型
- 通过高德地图进行地理编码和逆地理编码
- 支持地址排序和删除（软删除）

## 六、设计思路与亮点

### 6.1 审核驱动的车辆管理

系统采用**审核表驱动**的设计，车辆信息不直接操作车辆表，而是通过审核表作为中间层：

- **新增车辆**：写入审核表 → 审核通过 → 创建车辆表记录
- **编辑车辆**：更新审核表数据 → 重置为待审核 → 审核通过 → 更新车辆表记录
- **删除车辆**：审核表状态置为已删除（软删除）

这种设计确保了：
1. 所有车辆变更都有审核记录可追溯
2. 审核日志表记录每次状态变更，形成完整操作链
3. 驳回后用户可修改重新提交，不影响原车辆数据

### 6.2 双状态用户体系

用户表设计了 `status`（用户状态）和 `driver_status`（司机状态）两个独立字段：

- `status`：用户账号级别的封禁
- `driver_status`：司机角色级别的封禁

封禁司机时，仅影响司机相关功能（车辆管理），不影响用户基本功能。

### 6.3 电子围栏的布尔运算

电子围栏功能采用**蓝色围栏（待确认）+ 绿色围栏（已确认）**的双层设计：

- 蓝色围栏：从高德行政区划获取的边界数据，作为操作素材
- 绿色围栏：最终确认的服务区域，支持多边形含孔洞
- 使用 Clipper.js 进行布尔运算：
  - **加入围栏**：蓝色 ∪ 绿色 = 并集
  - **删除围栏**：绿色 - 蓝色 = 差集

### 6.4 前端轻量化

用户端和管理端均采用原生 HTML/CSS/JS 实现，无框架依赖：
- 使用 ES6 Module 进行模块化
- 公共样式和组件复用（common/style.css, common/ui.js）
- 按页面拆分，每个页面独立 HTML + JS + CSS
