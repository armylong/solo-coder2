# 调研选型 - 抢单行程归组

## 一、问题

拼拼坐的订单是预约制的，散布在不同日期、不同时段。一个司机一天可能跑多趟，每趟拉不同的人。

滴滴的模式是：司机在线 → 来单 → 接单 → 跑，一个单就是一个行程，时序是线性的。

拼拼坐的模式是：订单是预约制的，散布在不同日期、不同时段，一个司机一天可能跑多趟，每趟拉不同的人。所以"行程"不是天然存在的，需要组织出来。

当前抢单池的分组设计（时间 > 方向）只是一个展示分组，并不是一个真正的"行程"概念。司机看到一堆单子，他还是得自己脑补"这几单我能凑一车跑一趟"。

**核心矛盾：抢单池展示的是"订单"，但司机决策的粒度是"行程"，两者之间缺一层组织。**

## 二、关键洞察：系统的核心价值是调度

拼拼坐的初心：把线下拼车公司的套路完全线上化，解决只靠电话沟通的弊端。

线上化能解决的小问题：
- 接人时没有定位找不到 → 地图导航
- 出发前不知道司机到哪了心里慌 → 实时位置
- 电话沟通效率低 → 系统自动通知

但这些都是"小帮助"，线下公司没有这些也能运转，只是体验差一点。

**线下公司真正的最大难点是调度。** 人脑计算不能最优的调度——谁跟谁凑一车、先接谁后接谁、这趟满了下一趟怎么安排，全靠调度员的经验和脑子记，单子一多就顾不过来。

**能更好地调度，才是司机和乘客最大的诉求。**

- 司机诉求：系统帮我组好行程，我不用自己费脑子算
- 乘客诉求：系统帮我最快凑满车，不用干等
- 线下公司痛点：调度靠人，规模上不去

所以系统的核心价值不是"线上化"，而是**智能调度**。线上化只是手段，调度才是目的。

**但要注意：调度是核心能力，用户感知到的是结果。**

- 乘客感知不到"调度好不好"，感知到的是"能不能走、等多久、贵不贵"
- 司机感知不到"调度好不好"，感知到的是"能不能挣到钱、麻不麻烦"
- 调度好 → 凑车快 → 乘客等的时间短；调度好 → 满载率高 → 司机挣得多

所以第一版的目标不是调度最优，而是**比人脑调度强一点就够了**。线下公司调度也不完美，照样能运转。

**我们对线下调度了解不够，需要深入调研调度员视角。** 目前只了解了司机视角，不知道调度员脑子里怎么组织的——先按时间排还是先按方向排？一天50单5个司机怎么分配？凑不满怎么处理？不知道这些，我们设计的分组逻辑可能是在重复发明轮子，甚至发明了个方轮子。下次调研要找调度员聊。

## 三、关键洞察：抢单会不会乱？

五一实地调研：当地拼车公司都是派单制，不会发到大群里抢，怕乱。

**结论：拼拼坐抢单不会乱，因为传统拼车公司怕乱的本质原因是工具不行，不是模式不行。**

微信群抢单会乱的原因：
- 司机A和司机B同时说"我接了"，谁算？→ 冲突
- 张三的单被接了，但李四的单还在群里，没人知道张三已经有人接了 → 信息不同步
- 司机挑肥拣瘦，好的单秒抢，差的单没人接 → 资源错配
- 两个司机可能抢了同方向的单，各跑各的 → 运力浪费

拼拼坐作为系统，能做的事微信群做不了：
1. **锁单**：司机A接了，别人就看不到了，不存在抢冲突
2. **实时状态**：哪个单有人接了、哪个单还空着，一目了然
3. **智能分组**：系统把订单按时间方向分组，司机看到的是"这趟能拉3个人"，不是零散的单子
4. **防浪费**：系统知道哪些单已经被哪些司机接了，不会出现两个司机跑同一条线各拉1个人的情况

**本质：微信群抢单 = 无协调的抢单，当然乱；系统抢单 = 有协调的抢单，不会乱。**

传统拼车公司选派单，不是因为抢单模式有问题，而是因为他们的工具（微信群）撑不起抢单。我们有了系统，抢单就不是问题了。

## 四、关键洞察：扁平系统 vs 权威调度

线下拼车公司有绝对权威：让司机接哪个单就得接哪个，司机没太多话语权，毕竟还得指望着公司吃饭，跟员工和老板似的。

我们的系统是扁平的：司机自主选择，系统只做推荐。这种模式最终做出的形态可能比线下更好，但**是不是最优路线也未可知**。

**线下拼车公司的商业模式：赚信息费，不赚车费。** 车费是司机直接收的，公司不管。

- 来回给了≥5个人：收30元信息费
- 来回给了<5个人：收20元信息费
- 凑不满一车也发车，乘客得罪不起
- 价格是固定的，乘客都知道，没得商量

所以公司的利益和司机的利益不完全一致：
- 公司：只要成单就收信息费，1个人也收20，5个人也收30，差别不大
- 司机：1个人跑一趟可能赔本，4个人跑一趟才挣钱
- 公司"凑不满也发车"，本质上是拿司机的利润去换乘客的满意度

**1个人赔本的单，司机怎么办？** 开放问题，待下次调研确认。猜测线下公司可能靠以下方式消化：
1. 司机按往返算账，不按单程——去1人+回4人=这趟还是赚的
2. 公司用好单补差单——今天赔的单明天补个好单，长期拉平
3. 1个人的单往后放，尽量等凑人
4. 实在没人接，劝乘客改时间

两种模式对比：

| | 线下拼车公司（权威调度） | 拼拼坐（扁平系统） |
|---|---|---|
| 调度效率 | 高，公司说了算，不用商量 | 低，司机可能挑单拒单 |
| 司机体验 | 被动服从，没有选择权 | 主动选择，有控制感 |
| 系统复杂度 | 低，人肉调度 | 高，算法调度 |
| 运力利用率 | 高，强制满载 | 可能低，好单抢完差单没人接 |
| 扩展性 | 差，靠人调度有上限 | 好，算法可扩展 |

**这是一个开放问题，不急着下结论。** 前期司机少的时候，扁平模式够用；如果后期出现运力浪费严重的情况，可能需要引入更强的调度权威（比如强制派单、拒单惩罚等）。但那是后话，现在先做最自然的形态。

## 五、关键洞察：能分组就能派单？

既然系统有分组能力，能把订单聚合成行程，那为什么不直接派单？

**分组能力和派单能力之间还差一步：系统知不知道司机想跑哪。**

- 分组是纯客观的：时间重叠、方向一致 → 这几单能凑一起
- 派单是主观的：张司机今天想跑高铁站，李司机今天想跑机场 → 同一组单子，该派给谁？

但抢单Tab已经设计了筛选条件（出发地、方向、日期），司机设了这些，系统就知道了司机的意图。分组+司机意图，确实就够派单了。

**所以真正的问题不是"能不能派单"，而是"派单和抢单对司机来说体验差多少"：**

| | 抢单（系统分组展示） | 派单（系统直接派） |
|---|---|---|
| 司机感受 | 我选的，我有控制感 | 系统安排的，我得服从 |
| 司机操作 | 浏览 → 挑选 → 接单 | 收到通知 → 接/拒 |
| 多司机竞争 | 先到先得，公平 | 系统决定给谁，需要调度算法 |
| 适合阶段 | 前期单少，司机少 | 后期单多，司机多 |

**但两者的边界其实很模糊。** 想想这个场景：

> 系统分好组了，推了一个通知给匹配的司机："8:00-8:20 高铁站方向3人，要不要接？"

这到底是抢单还是派单？司机视角就是"系统给我推了一趟活，我接不接"。

**分组+推送 ≈ 软派单。**

**结论：不用纠结抢单还是派单，做的是"推荐行程+推送通知"。前期司机少就是抢单（谁先接算谁的），后期司机多了加调度规则就自然过渡成派单。产品形态是一样的，区别只在调度规则。**

## 六、关键洞察：座位公里利用率——调度的核心指标

从根源推导，不依赖线下经验，调度的核心目标是什么？

拼车的本质：一辆车从A出发，经过若干上下车点，最终到达某个终点。

- 车跑了总里程 D
- 乘客坐了各自的里程 d1, d2, d3...
- 乘客付费 = d1×单价 + d2×单价 + d3×单价
- 司机成本 ≈ D×每公里成本

**核心指标：座位公里利用率 = 乘客总座位公里 / 车总行驶公里**

举例：

| 场景 | 利用率 | 司机感受 |
|------|--------|---------|
| 4个人都去高铁站，全程50公里 | 4×50/50 = 400% | 最赚 |
| 3个去高铁站(50公里) + 1个去A市(再走30公里) | (3×50+1×80)/80 = 287% | 还行 |
| 1个去A市 + 1个去B市(反方向) | ≈100% | 赔本 |

**最大化座位公里利用率，就能推导出所有调度规则：**

1. 同目的地优先凑一车 → 利用率最高，0绕路
2. 顺路的可以混坐 → 绕路少，利用率还行
3. 不顺路的坚决不凑 → 绕路太多，利用率暴跌
4. 凑不满就少派车 → 9人3司机要3+3+3，不要4+4+1

**线下公司摸索出来的规则，全部是这条原则的近似解。** 他们不知道"座位公里利用率"这个概念，但经验把他们导向了同样的结论。

**这个指标还能推导出线下经验覆盖不到的东西：** 比如定价——混坐时绕路远的乘客应该多付钱，因为占用了司机额外的公里数。

## 七、关键洞察：分组策略修正——路线优于方向

五一调研发现，线下实际情况是：

- 市里和高铁混坐是常态，纯高铁一车反而是少数
- 司机先送市里再送高铁，因为市里比较绕
- 市里35元，高铁45元，同车不同价
- 方向不是分组维度，方向只是定价维度

但"方向不重要"这个结论是错的。准确地说：

- 市里和高铁站：差几公里，顺路，能混，只是价格不同
- A市和B市：差几十上百公里，不顺路，不能混

**关键不是"同不同方向"，而是"路线差多远"。** "方向"应改为"路线"：

- 如果去B市必须经过A市的高铁站，那他们坐一起也不是不可以
- 但会让司机利益受损：A市下了3个人，基本等于空车到B市
- 所以顺路混坐只是兜底策略，同目的地优先才是正道

**修正后的分组优先级：时间 > 目的地 > 顺路程度**

| 优先级 | 策略 | 利用率 | 例子 |
|-------|------|--------|------|
| 最高 | 同时间+同目的地 | 400% | 4个去高铁站的 |
| 中等 | 同时间+顺路目的地 | 287% | 3个去高铁站+1个去A市(经过高铁站) |
| 最低 | 同时间+不顺路 | ≈100% | 3个去A市+1个去B市，只作为兜底 |

**顺路程度可以量化：共同路段 / 总路段。** 共同路段占比越高，混坐越合理。

**顺路不是看直线距离，是看实际路线。** 市里和高铁站在一条线上，经过市里去高铁站不算绕；但如果市里和高铁站是两个方向，绕过去就要额外跑30公里，利用率完全不同。

## 八、关键洞察：负载均衡——少派车优于均匀分配

线下调度的另一个原则：**在乘客和司机之间尽量均匀分配，避免出现极端情况。**

```
10个乘客3个司机：
✅ 4+3+3
❌ 4+4+2（2个人的司机利用率太低）

9个乘客3个司机：
✅ 3+3+3
❌ 4+4+1（1个人的司机赔本）

9个乘客4个司机：
✅ 3+3+3+先不派第4个
❌ 3+3+2+1（1个人的司机赔本，2个人的也勉强）
```

**原则：宁可少派车，也不要让某个司机利用率过低。** 这也是座位公里利用率的推论——少派一辆车，总行驶里程少，整体利用率更高。

## 九、关键洞察：调度是持续迭代的，不是一次性的

线下公司的调度流程：

1. **先粗排**：根据明天的预约单，按时间+目的地粗排一遍
2. **再微调**：第二天有新单进来，往现有的分组里塞，尽量往合理的方向调整
3. **实时应变**：出发前还有变化（取消、加单），调度员随时调整

**我们的系统也应该这样：调度不是一次性的计算，是持续迭代的过程。** 每次有新订单进来，都重新计算分组，尽量保持现有分组稳定的同时，把新订单归入最合适的位置。

## 十、待讨论问题

### 1. "行程"是不是一等实体？

**问题**：系统里只有"订单"（乘客视角：我要从A到B），没有"行程"（司机视角：我某天某时跑一趟A→B，拉了这几个人）。订单和行程需要对接，但它们不是同一个东西。

如果行程是一等实体，意味着：
- 行程有独立的数据模型和生命周期
- 订单通过关联行程来归属
- 司机接单 = 接下某个行程，行程下的订单自动锁定

如果行程不是一等实体，意味着：
- 行程只是订单的展示分组，没有独立状态
- 司机接单 = 逐个接订单，系统靠规则判断哪些订单能凑一起
- 灵活但容易乱

### 2. 系统预组 vs 司机自建

**方案A：系统预组行程，司机选行程**

系统自动把订单聚合成"候选行程"，司机看到的是一个个现成的行程包：

```
📋 5月12日 候选行程

🚗 行程1  08:00-08:30 小区→高铁站  3/4人
   张三 08:00 ×1  李四 08:10 ×1  王五 08:15 ×1
   [接下这趟]

🚗 行程2  09:30-10:00 高铁站→小区  2/4人
   赵六 09:30 ×1  孙七 09:40 ×1
   [接下这趟]

🚗 行程3  14:00-14:30 小区→机场  1/4人
   周八 14:00 ×1
   [接下这趟]
```

- 优点：司机决策简单，看行程接行程
- 缺点：系统分组逻辑可能不准；6个人同方向但车只能坐4个，谁上谁不上？司机想微调不方便

**方案B：司机建行程，系统推荐订单**

司机先声明"我要跑一趟"，系统推荐匹配的订单，司机挑选：

```
第一步：司机建行程
┌─────────────────────────────┐
│  🚗 我要跑一趟               │
│  📅 5月12日                  │
│  📍 从：家                   │
│  🏁 往：高铁站方向            │
│  🕐 出发：8:00-8:30          │
│  💺 座位：4                   │
│           [查看匹配订单]       │
└─────────────────────────────┘

第二步：系统推荐匹配订单，司机勾选
┌─────────────────────────────┐
│  ✅ 张三 8:00 ×1人            │
│  ✅ 李四 8:10 ×1人            │
│  ✅ 王五 8:15 ×1人            │
│  ☐  赵六 8:20 ×1人 (稍远)    │
│  ☐  孙七 8:25 ×2人 (超载)    │
│           [确认行程 3/4人]     │
└─────────────────────────────┘
```

- 优点：司机主导，灵活度高，不会出现系统硬塞不合适的组合
- 缺点：操作步骤多，司机需要主动规划

**方案C：A+B 混合**

系统预组行程作为推荐，司机可以微调：

1. 抢单池默认展示系统预组的候选行程（方案A的体验）
2. 司机点进某个候选行程后，可以增减订单（方案B的灵活性）
3. 司机也可以从零建行程，系统推荐匹配订单

```
抢单池（默认视图 - 系统预组）
┌─────────────────────────────┐
│  🕐 8:00-8:20 高铁站方向 3人  │  ← 点进去可以微调
│  🕐 9:30-10:00 市区方向 2人   │
│  🕐 14:00-14:30 机场方向 1人  │
│                             │
│  ➕ 自己组一趟                │  ← 从零建行程
└─────────────────────────────┘
```

- 优点：兼顾简洁和灵活
- 缺点：开发复杂度最高

### 3. 按时到达的时间换算

之前文档已确定：不管用户选"按时出发"还是"按时到达"，系统内部统一换算成**出发时间窗口**。

换算公式：`出发窗口 = 到达时间 - 路程耗时 - 弹性提前量`

示例：

| 用户选的 | 换算成出发窗口 | 匹配优先级 |
|---------|-------------|----------|
| 按时出发 14:00-14:20 | 14:00-14:20 出发 | 时间窄，优先匹配 |
| 按时出发 14:00-17:00 | 14:00-17:00 出发 | 时间宽，灵活拼 |
| 按时到达 13:10-13:40 | 约12:20-13:00 出发 | 时间窄，必须早出发 |

**待解决：路程耗时怎么算？**
- ~~方案1：调地图API实时计算（准但依赖外部服务）~~
- ~~方案2：用历史数据估算（离线但不够准）~~
- ~~方案3：常用路线预计算缓存（折中）~~

**结论：历史数据不可靠，线下也估不准。**

今天高速封路、明天下雨、后天节假日堵车，路程耗时波动太大，历史数据没有参考价值。线下拼车公司靠经验，估不准的时候就使劲把时间拉长，做好冗余。

**我们的策略：不做精确估算，做冗余。**

- 按时到达的乘客：出发窗口 = 到达时间 - 基础耗时(保守值) - 大幅冗余
- 冗余宁可大不可小：早到1小时乘客能接受，晚到10分钟乘客要投诉
- 基础耗时用地图API的"最慢预估"而非"平均预估"
- 常用路线（小区→高铁站）可以人工标定一个保守耗时，比算法靠谱

### 4. 调度的核心难点：聚类

把出发时间极度接近、目的地极度接近的人凑一车，本质是一个**聚类问题**，而且约束条件互相矛盾：

- 时间越接近越好 → 但等太久乘客不满意
- 目的地越接近越好 → 但绕一点路也能接受
- 座位不能超 → 但留空座又浪费运力

这不是简单的排序或分组，需要在多个约束之间找最优解。前期订单量小，简单规则够用；后期需要正经的调度算法。

### 5. 按座位数攒人头

分组时座位数是硬约束：4座车最多4人，6座车最多6人。

**问题：系统分组时还不知道哪个司机会接，怎么知道座位数？**

- 方案1：按默认4座分组（大多数车是4座），6座司机接单后看到两组可合并
- 方案2：按最小座位数分组（保守），避免任何车型接不了
- 方案3：分组时不考虑座位数，只按时间方向分，座位数在司机接单时作为筛选条件

**倾向方案1**：4座是主流，按4座分组最自然。6座司机是加分项，能看到更多可合并的行程。

### 6. 已有行程来了新订单，自动塞还是推送确认？

分两种情况：

**行程还没司机接（候选行程）**：
- 新订单直接归入匹配的候选行程，不需要确认
- 因为行程本身还没定，分组随时可以调整

**行程已有司机接（确认行程）**：
- 新订单匹配到了这个行程 → 推通知让司机确认，不自动塞
- 原因：司机可能觉得够了不想再绕路，自动塞会侵犯司机的控制感
- 司机确认后，订单锁定到行程；司机拒绝，订单回到抢单池

```
司机收到通知：
┌─────────────────────────────┐
│  📢 新订单匹配到您的行程       │
│                             │
│  行程：5月12日 08:00 小区→高铁站 │
│  当前：3/4人                  │
│                             │
│  赵六 08:10 ×1人 高铁站方向   │
│                             │
│  [加入行程]  [不需要]          │
└─────────────────────────────┘
```

## 十一、调度漏斗（草案，待验证）

订单进来后，按以下步骤逐步分组：

**1. 按时间窗口分组**
同一时间段出发的人归到一起。时间是最硬的约束，时间对不上其他都白搭。

**2. 按目的地分组**
在同一时间段内，同目的地的人优先凑一车。利用率最高，0绕路。

**3. 按路线合并**
不同目的地但顺路的，看能不能合并。比如3个去高铁站+1个去A市（经过高铁站），合并成一车，利用率287%还行。顺路程度 = 共同路段 / 总路段，占比越高越值得合并。

**4. 按座位数切分**
合并后超过4人的，按4人上限切分成多趟。比如7个去高铁站的 → 4+3，不会4+2+1。

**5. 按负载均衡分配司机**
多趟行程多个司机时，尽量均匀分配。9人3司机 → 3+3+3，不是4+4+1。宁可少派车，也不要让某个司机利用率过低。

**6. 处理剩单**
走完前5步还剩的零散订单（1个人去冷门目的地），单独挂着，等后续新订单来了再重新走漏斗。

**待验证问题：步骤3和步骤4的先后关系。**

比如8:00-8:20有5个去高铁站的+1个去A市的：
- 先合并再切分：4个高铁+1个高铁+A市混坐
- 先切分再合并：4个高铁单独一车，1个高铁+A市混坐另一车

步骤3和4可能不是严格的先后关系，而是要交替考虑。
