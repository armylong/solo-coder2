# 调研选型 - 拼车派单方案

## 一、背景

用户可以发起拼车订单，选择出发地、目的地、出行时间、人数，系统根据用户数据匹配司机。需要确定派单模式。

## 二、用户下单时间选择

用户下单时选择出行时间，有两种模式：

1. **按时出发**：选择出发时间范围，如 `日14:00 - 14:20`
   - 不着急的乘客，时间窗口可以很宽，如 `日14:00 - 17:00`
   - 时间窗口窄的乘客等不了太久

2. **按时到达**：选择到达时间范围，如 `日13:10 - 13:40`
   - 着急的乘客，如赶高铁，耽误了要赔钱
   - 这类乘客早到2小时都能接受，只要别晚了

### 时间换算

不管用户选"按时出发"还是"按时到达"，系统内部都可以换算成**出发时间窗口**，用于统一匹配逻辑：

| 用户选的 | 换算成出发窗口 | 匹配优先级 |
|---------|-------------|----------|
| 按时出发 14:00-14:20 | 14:00-14:20 出发 | 时间窄，优先匹配 |
| 按时出发 14:00-17:00 | 14:00-17:00 出发 | 时间宽，灵活拼 |
| 按时到达 13:10-13:40 | 约12:20-13:00 出发 | 时间窄，必须早出发 |

### 线下实际情况

- 拼车公司会忽悠乘客早点出发，为了更多容错率（路上要接一圈人）
- 实在没法早的，以该乘客时间为准，去协调其他乘客
- 司机自己估算时间，基本不会耽误事
- 用户一般提前一天预订，提前一两小时也能订到但有概率订不到
- 提前半小时着急走只能看运气，不舍得包车大概率按时走不了

## 三、派单方案调研

### 方案1：直接派单制

**流程**：用户下单 → 系统匹配司机 → 凑满人 → 通知司机发车

**司机端配置**：
- 选择运营路线（A→B）
- 在路线范围内画几个小圈，表示自己通常活动的区域
- 设定上班时间范围 + 每天接单数

**匹配逻辑**：
- 用户经纬度落在司机的圈内 → 找到候选司机
- 同一出发时间窗口内的乘客凑在一起

**优点**：
- 乘客体验好，系统自动安排
- 平台整合资源后，客源合并，凑人成功率高
- 对乘客的好处：不用跟着司机转悠接人、时间差不多的乘客更好凑、不用被忽悠等半天

**缺点**：
- 凑人算法复杂：4人出发时间不一致怎么办、绕路顺序怎么排、临时取消怎么办
- 凑不满4人时：问乘客包不包车，不包就告诉拼不上
- 异常处理多：取消、迟到、堵车等

**异常处理**：
- 第4个乘客取消 → 司机端提示有乘客被取消，司机可以自己决定出发（点击出发锁单），也可以等系统重新组人

### 方案2：司机订阅制

**流程**：用户下单 → 系统推送订单给订阅了该路线的司机 → 司机自己抢单

**司机端配置**：
- 选择订阅路线
- 系统把对应路线的乘客信息推送到列表

**优点**：
- 实现简单，跟现有运营路线架构契合
- 司机自主决策，时间怎么凑司机自己判断（比系统更靠谱）
- 跟现有本地拼车模式一致（拼车公司把用户信息发到微信群，司机自己抢单）

**缺点**：
- 可能没人接单，用户体验不好
- 多个司机抢同一单需要处理

### 方案3：轮流排队制

司机按上线顺序排队，来单了按顺序派。类似机场出租车排队。

**不适用原因**：本地拼车不是随时在线等单的模式，司机有自己的时间安排，排队不现实。

### 方案4：乘客选司机制

系统把附近可用司机推给乘客，乘客自己选。类似滴滴早期。

**不适用原因**：拼车场景乘客不关心谁开，只关心能不能走、几点走，选司机反而增加决策负担。

### 方案5：固定班次制

像班车一样，设定固定发车时间，乘客选班次买票，到点发车。

**不适用原因**：本地拼车凑齐了就走，凑不齐就等，没有固定班次的概念。

### 方案6：拍卖/竞价制

订单发出来，司机出价抢单。货运平台常用。

**不适用原因**：拼车价格应该是固定的，不适合竞价。

### 方案7：社群制

微信群的数字化版——订单发到群里，司机看到就接。

**本质**：就是方案2的抢单池，换个形式。

## 四、最终决定：派单 + 抢单

派单和抢单不是对立的，是优先级的关系：

1. 系统发现能凑满 → 直接派单给司机（优先）
2. 司机10分钟没接 → 降级为抢单（兜底）
3. 凑不满 → 直接进抢单池

### 司机端设计

两个 tab：
- **待接单**：系统派来的已凑满的订单
- **抢单池**：所有未拼满的订单

司机在任何 tab 接了单，就锁住这几个乘客。

司机也可以放弃派单，放弃后订单进入抢单池。

### 分阶段策略

| 阶段 | 用户量 | 策略 |
|------|--------|------|
| 前期 | 少 | 抢单为主，订单少系统凑人成功率低 |
| 后期 | 多 | 派单为主，订单密度够了系统自动凑人 |

方案2是基础能力，方案1是增强能力，用户量到了自然过渡。

### 核心原则

- 能挣就挣，挣不到拉到，不强行搞
- 拼不满车就问包不包车，不包就告诉拼不上
- 司机可以放弃派单，不强制

### 抢单池分组策略

抢单池不是简单的时间线列表，而是系统自动分组，方便司机一眼看出哪些人能凑一车。

**分组优先级**：时间 > 方向

- **时间是最硬的约束**：时间对不上，方向再一致也坐不了一车
- **方向是软约束**：市里和高铁站的也可以坐一车，只是定价不同，各自给司机扫钱就行
- **优化目标**：在促成交易的基础上，尽量把同方向的乘客分在一组，减少司机绕路

**分组维度**：

1. 按时间分组：同一出发时间窗口内的乘客归为一组
2. 按方向优化：在时间组内，尽量把同方向的乘客放一起

**示例**：

```
🕐 8:00-8:20 出发（4人）← 凑满了，可发车
  📍 高铁站方向（3人）
    - 张三 8:00出发 ×1人
    - 李四 8:10出发 ×1人  
    - 王五 8:15出发 ×1人
  📍 市区方向（1人）
    - 赵六 8:10出发 ×1人

🕐 9:00-9:30 出发（2人）
  📍 高铁站方向（2人）
    - 孙七 9:00出发 ×1人
    - 周八 9:20出发 ×1人
```

**平台价值**：线下拼车最烦的就是接到的人东一个西一个，绕来绕去。平台用算法优化分组，让司机接到的单尽量同方向，这就是平台的核心价值之一。
