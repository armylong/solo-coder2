# 调研选型 - 派单抢单

## 一、核心逻辑

派单优先，抢单兜底：

1. 用户下单 → 系统尝试凑人 → 凑满了直接派单给司机
2. 司机10分钟没接 → 降级进抢单池
3. 凑不满 → 直接进抢单池

分阶段策略：

| 阶段 | 用户量 | 策略 |
|------|--------|------|
| 前期 | 少 | 抢单为主，订单少系统凑人成功率低 |
| 后期 | 多 | 派单为主，订单密度够了系统自动凑人 |

核心原则：
- 能挣就挣，挣不到拉到，不强行搞
- 拼不满车就问包不包车，不包就告诉拼不上
- 司机可以放弃派单，不强制

## 二、待讨论问题

### 1. 司机位置与推单范围

**问题**：系统根据什么推单给司机？

**方案A：根据常用地址推送**
- 司机在"我的"里设置常驻地址（家、公司），系统根据常驻地址推单
- 优点：设置一次就行，省心
- 缺点：太死板，司机今天想跑高铁站明天想跑机场，常用地址推不准

**方案B：司机在抢单tab设置筛选项**
- 司机在抢单tab顶部设置出发地、方向、日期，系统根据筛选条件展示
- 优点：司机最清楚自己想跑哪，灵活
- 缺点：每次都要选

**方案C：常用地址预填 + 司机可改**
- 出发地默认是常用地址（家），司机可随时切换
- 方向和日期由司机选择
- 兼顾便捷和灵活

**倾向方案C**，具体设计：

```
┌─────────────────────────────┐
│  🔍 抢单池                   │
│                             │
│  📍 出发：家（可切换）        │
│  🏁 方向：全部 ▾             │
│  📅 日期：今天 ▾             │
├─────────────────────────────┤
│  🕐 8:00-8:20 高铁站方向 3人 │
│  张三×1 李四×1 王五×1       │
│              [接单]          │
├─────────────────────────────┤
│  🕐 9:00-9:30 高铁站方向 2人 │
│  孙七×1 周八×1              │
│              [接单]          │
└─────────────────────────────┘
```

- 出发地：默认常用地址（家），可切换
- 方向：全部 / 高铁站 / 机场 / 市区 等
- 日期：今天 / 明天 / 后天
- 司机改了筛选条件，抢单池实时刷新

**注意**：web页面无法后台持续上报位置，所以不能用实时定位，只能用地址。

### 2. 司机上班/请假

**问题**：司机计划哪天上班、哪天请假，从哪设置？

**方案A：默认上班，请假标记**
- 司机注册后默认每天都上班，系统每天都推单
- 请假时标记哪天休息，那天不推单
- 适合自由职业模式，想接就接

**方案B：排班制**
- 司机设置每周哪几天上班
- 太复杂，不适合拼车场景

**倾向方案A**，具体设计：

```
┌─────────────────────────────┐
│  🔍 抢单池                   │
│  📅 今天 ✓上班  明天 ✓上班  后天 ☐休息 │
├─────────────────────────────┤
│  ...                        │
└─────────────────────────────┘
```

- 抢单tab顶部放日期开关，一键切换
- 休息日不推单，但司机随时可以切回上班
- 简单直接，不需要复杂的排班系统

## 三、已确定的设计

### 抢单池分组策略

抢单池按时间+方向自动分组，方便司机一眼看出哪些人能凑一车。

分组优先级：时间 > 方向

- 时间是最硬的约束：时间对不上，方向再一致也坐不了一车
- 方向是软约束：市里和高铁站的也可以坐一车，只是定价不同
- 优化目标：在促成交易的基础上，尽量把同方向的乘客分在一组，减少司机绕路

分组维度：
1. 按时间分组：同一出发时间窗口内的乘客归为一组
2. 按方向优化：在时间组内，尽量把同方向的乘客放一起

示例：

```
🕐 8:00-8:20 出发（4人）← 凑满了，可发车
  📍 高铁站方向（3人）
    - 张三 8:00出发 ×1人
    - 李四 8:10出发 ×1人
    - 王五 8:15出发 ×1人
  📍 市区方向（1人）
    - 赵六 8:10出发 ×1人

🕐 9:00-9:30 出发（2人）
  📍 高铁站方向（2人）
    - 孙七 9:00出发 ×1人
    - 周八 9:20出发 ×1人
```

### 司机端Tab结构

- 拼车Tab：当前行程/订单表单 + 未来行程
- 抢单Tab：抢单池（司机专属）
- 我的Tab：个人信息、车辆管理、历史行程、收入统计

### 时间换算

不管用户选"按时出发"还是"按时到达"，系统内部统一换算成出发时间窗口：

| 用户选的 | 换算成出发窗口 | 匹配优先级 |
|---------|-------------|----------|
| 按时出发 14:00-14:20 | 14:00-14:20 出发 | 时间窄，优先匹配 |
| 按时出发 14:00-17:00 | 14:00-17:00 出发 | 时间宽，灵活拼 |
| 按时到达 13:10-13:40 | 约12:20-13:00 出发 | 时间窄，必须早出发 |

### 异常处理

- 司机10分钟没接派单 → 降级进抢单池
- 司机放弃派单 → 订单进入抢单池
- 乘客取消 → 司机端提示，司机决定继续出发还是等系统重新组人
