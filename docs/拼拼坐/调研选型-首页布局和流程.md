# 调研选型 - 首页布局和流程

## 一、首页布局

**决定：统一首页结构，司机与用户共享**

```
┌─────────────────────────────┐
│                             │
│  主卡片（当前行程/订单表单）  │
│                             │
└─────────────────────────────┘

  📋 未来行程
  ┌───────────────────────┐
  │ 明天08:00 学校→机场 1/4人 │
  └───────────────────────┘
  ┌───────────────────────┐
  │ 后天14:00 商场→火车站 2/4人│
  └───────────────────────┘
```

- **主卡片**：当前最紧急的行程（正在路上 > 即将出发 > 匹配中），没有行程时就是订单表单
- **未来行程**：已确认但还没开始的行程列表，点击可切换到主卡片展示详情
- **历史行程**：放在"我的"页面，不放首页（低频操作，首页保持简洁）
- 司机和用户完全一样，只是卡片内容根据角色不同展示不同信息
- 用户：看到司机信息
- 司机：看到所有乘客信息

新用户/无行程时：首页就一个订单表单卡片，干净利落，直接引导拼车。未来有需要可在主卡片下方加广告横幅。

第一版不做常用路线快捷入口，等订单功能做好后再加。

## 二、乘客与司机视图

**决定：统一入口，角色在流程中自然区分**

- 同一套 UI，司机也可以当乘客
- 角色差异在具体操作中体现（如"发拼车"按钮，司机点=发行程，乘客点=发需求）
- 判断依据：ppz_user.car_count > 0

## 四、核心流程优先级

**决定：先做乘客发需求 → 司机抢单**

- 前期用户少，司机主动发行程没人拼，乘客先有需求司机才去接，更容易冷启动
- 派单能力后续再加（参考：调研选型-拼车派单方案.md）

## 五、订单状态流

### 正常流

```
我要拼车 → 匹配司机 → 司机已接单 → 已电话联系确认 → 司机在赶来的路上 → 行程中 → 已到达 → 已完成
```

各状态说明：
- **我要拼车**：用户填写出发地+目的地+时间+人数
- **匹配司机**：系统派单/抢单中，卡片边框彩色旋转
- **司机已接单**：司机抢单成功，但尚未联系用户
- **已电话联系确认**：司机电话联系用户确认，锁单
- **司机在赶来的路上**：司机点击"出发接人"，用户点卡片可看司机实时位置
- **行程中**：已上车出发
- **已到达**：到达目的地
- **已完成**：订单完成，卡片回到表单，订单挪到历史列表（下方有评价入口）

### 异常流

- **匹配失败**：无人接单，直接告知用户拼不上
- **已取消**：
  - 用户取消：回到表单第一项重新填写
  - 司机取消：司机必须处理（点击确认）才能接下一单

## 六、填写流程

卡片内左移动画，每步有上一步/下一步按钮：

1. 出发地 + 目的地
2. 时间选择（日期 + 时间 + 弹性范围）
3. 人数选择（1-4人）

## 七、卡片各状态设计

### 1. 初始状态 - 填写出发地目的地

```
┌─────────────────────────────┐
│                             │
│  📍 从哪出发？              │
│  ─────────────────          │
│  🏁 要去哪？                │
│  ─────────────────          │
│                             │
│              [下一步 →]     │
└─────────────────────────────┘
```

### 2. 填写时间

卡片展示：
```
┌─────────────────────────────┐
│                             │
│  按时出发  ▾                │
│                             │
│  🕐 明天 14:00              │
│  ⏱ 最多等30分钟             │
│                             │
│  [← 上一步]    [下一步 →]   │
└─────────────────────────────┘
```

点击时间行 → 底部弹出三列滚轮选择器：
```
┌─────────────────────────────────┐
│  选择出发时间              [确定] │
│                                 │
│  ┌──────┐  ┌────┐  ┌────┐      │
│  │ 今天  │  │ 13 │  │ 55 │      │
│  │ 明天  │  │ 14 │  │ 00 │ ←选中│
│  │ 后天  │  │ 15 │  │ 05 │      │
│  │ 周四  │  └────┘  └────┘      │
│  │ 周五  │   小时     分钟       │
│  │ 周六  │                      │
│  │ 周日  │                      │
│  └──────┘                       │
│   日期(未来7天)                  │
└─────────────────────────────────┘
```

点击弹性行 → 底部弹出按钮组选择：
```
按时出发时：
⏱ 最多等30分钟
[10分] [20分] [30分] [1小时]

按时到达时：
⏱ 可提前30分钟
[10分] [20分] [30分] [1小时]
```

- 日期列：今天/明天/后天/周x，显示未来7天
- 小时列：0-23 滚动
- 分钟列：0-59 滚动
- 按时出发/按时到达可切换（按时到达时标题改为"选择到达时间"，弹性行文案改为"可提前"）
- 弹性方向：
  - 按时出发14:00 + 最多等30分 = 14:00-14:30之间出发
  - 按时到达14:00 + 可提前30分 = 13:30-14:00之间到达（换算成出发时间窗口）
  - 按时出发只能往后延，按时到达只能往前提，绝不反向

### 3. 填写人数

```
┌─────────────────────────────┐
│                             │
│  👤 几个人乘车               │
│                             │
│  [1人] [2人] [3人] [4人] [5人][6人] │
│                             │
│  ☐ 包车（独享整车）          │
│                             │
│  [← 上一步]    [确认拼车]   │
└─────────────────────────────┘
```

- 人数1-6人，覆盖5座车（4乘客位）和7座车（6乘客位）
- 包车：勾选后司机不能再接其他乘客，匹配到的车全部座位归你
- 非包车：司机可以继续拼其他人
- 数据库需加 is_charter 布尔字段
- 匹配逻辑：按人数优先匹配座位够的车，5座优先7座兜底

### 4. 匹配司机（边框彩色旋转）

```
╔═════════════════════════════╗  ← 彩色旋转边框
║  📍 小区  →  🏁 高铁站      ║
║  🕐 明天14:00 最多等30分 2人包车║
║                             ║
║  🔄 正在匹配司机...          ║
║                     [取消]  ║
╚═════════════════════════════╝
```

- 基础信息区：出发地→目的地 + 时间 + 最多等/可提前 + 人数 + 包车标签（有则显示）
- 进度状态区：只展示当前状态文字，不做节点进度条
- 旋转边框：只在匹配阶段旋转，匹配到后变静态主题色边框
- 司机取消后自动重新匹配，用户无感知，状态文字始终显示"正在匹配司机..."
- 取消：随时可取消，取消后卡片回到初始表单


- 查看位置：乘客在车上，获取自身GPS定位即可，展示当前位置+路线，看还有多久到
- 不需要联系司机（都在车上）
- 不可取消
- 司机或乘客任一方点击"确认到达"流转到卡片9

司机信息递进规则：
- 司机已接单：只展示姓氏（张师傅）
- 1小时未联系：展示姓氏 + 联系电话按钮
- 已电话联系确认：姓氏 + 车牌号 + 联系电话
- 司机在赶来的路上/行程中：姓氏 + 车颜色+型号 + 车牌号 + 联系电话 + 查看位置
- 永远不展示几座车信息，避免7座车偏见

- 司机或乘客任一方点击"确认到达"进入此状态
- 到达=完成，不需要额外确认步骤
- 评价打分：点击进入评价页面
- 卡片停留在已到达状态，刷新页面后订单进入历史列表，卡片恢复为初始表单

### 异常：匹配失败

```
┌─────────────────────────────┐
│  📍 小区  →  🏁 高铁站      │
│  🕐 明天14:00 最多等30分 2人包车│
│                             │
│  😔 暂时拼不上车             │
│              [重新填写]      │
└─────────────────────────────┘
```

- 触发条件：出发时间已过，无人接单，不可能再拼
- 重新填写：保留出发地、目的地、人数，时间需重新选择（已过期），跳转到卡片1从头开始

## 八、司机端卡片各状态设计

核心原则：司机和用户共享同一个首页结构——一个主卡片。角色决定展示内容。
- 用户：1个行程卡片，看到司机信息
- 司机：1个行程卡片，看到所有乘客信息
- 司机不给乘客评价打分（与主流打车软件一致，避免报复性差评）
- 统一到达：司机或乘客任一方点击"确认到达"即可，不需要依次到达

### 异常：已取消

用户取消（匹配中/已接单/已确认）：
```
┌─────────────────────────────┐
│  📍 小区  →  🏁 高铁站      │
│  🕐 明天14:00 最多等30分 2人包车│
│                             │
│  ❌ 已取消                   │
│              [重新填写]      │
└─────────────────────────────┘
```

- 用户取消：匹配中、已接单可直接取消；已电话联系确认后需先联系司机才能取消
- 已电话联系确认后取消：点击取消先弹窗提示"请先联系司机说明情况"，带[联系司机]按钮
- 司机在赶来的路上/行程中：不可取消
- 司机取消：不展示此卡片，自动回到卡片4"司机已取消，正在重新匹配..."
- 重新填写：保留出发地、目的地、人数，时间需重新选择（已过期），跳转到卡片1从头开始
