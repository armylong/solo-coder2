# 用户认证系统详细设计

## 一、Token 机制

系统使用双 Token 设计：JWT + 随机 Token。

### 1.1 JWT Token

JWT 用于快速验证 Token 有效性，无需查库。

```go
type Claims struct {
    UID        int64  `json:"uid"`
    Name       string `json:"name"`
    DeviceType string `json:"device_type"`
    jwt.RegisteredClaims
}
```

- 签名算法：HS256
- 密钥：通过 `JWTConfig.Secret` 配置
- 有效期：7 天（由调用方决定）
- 用途：中间件第一层校验，快速拒绝无效/过期 Token

### 1.2 随机 Token

随机 Token 是实际存储和传递的凭证。

```
格式：hex 编码的 32 字节随机数
示例：a3f1b2c4d5e6f7890123456789abcdef0123456789abcdef0123456789abcdef
```

- 存储在 `tb_user_token` 表
- 缓存在 `SyncMapCache` 中
- 客户端通过 `Authorization` Header 传递

### 1.3 为什么用双 Token

| | JWT | 随机 Token |
|---|---|---|
| 验证方式 | 签名验证，无需查库 | 查数据库/缓存 |
| 能否主动失效 | 不能 | 能（删记录/清缓存） |
| 用途 | 快速过滤无效请求 | 实际会话凭证 |

中间件流程：先验证 JWT 签名（快速拒绝伪造 Token），再通过随机 Token 加载用户（支持主动失效）。

## 二、缓存设计

### 2.1 数据结构

```go
type SyncMapCache[T any] struct {
    cache sync.Map
}

// 存储 key 格式：login_{token}
// 存储 value：*LoginUserInfo（泛型 T）
```

### 2.2 缓存操作

| 操作 | 方法 | 场景 |
|---|---|---|
| 写入 | `Set(token, user)` | 登录成功、中间件加载用户后回写 |
| 读取 | `Get(token)` | 中间件验证时优先查缓存 |
| 删除单个 | `Delete(token)` | 登出、踢下线 |
| 批量删除 | `DeleteByTokens(tokens)` | 踢下线（按设备类型） |
| 按 UID 清空 | `ClearByUID(uid, getUID)` | 管理员踢人（清空该用户所有设备） |

### 2.3 缓存一致性

```
登录：  写数据库 → 写缓存
验证：  读缓存 → [未命中] 读数据库 → 写缓存
登出：  删数据库 → 删缓存
踢下线：删数据库 → 删缓存（可能批量）
改密码：删数据库 → 清空该 UID 所有缓存
```

缓存是数据库的加速层，不做持久化保证。服务重启后缓存为空，中间件会自动回源数据库。

## 三、中间件

### 3.1 Middleware — 登录验证

```
1. 从 Header 取 Authorization
2. 缺失 → 返回 401 "请先登录"
3. 解析 JWT → 失败 → 返回 401 "Token无效或已过期"
4. 查缓存 → 命中 → 设置上下文
5. 未命中 → UserLoader 查库 → 失败 → 返回 401 "请重新登录"
6. 查库成功 → 写缓存 → 设置上下文
7. ctx.Next()
```

### 3.2 RequirePermission — 权限检查

```go
func RequirePermission[T any](check func(T) bool, unauthorized func(*gin.Context, string)) gin.HandlerFunc
```

通用权限检查，通过 `check` 回调判断是否有权限。

应用层封装了两个具体实现：

```go
func RequireAdmin(ctx *gin.Context) {
    RequirePermission(func(u *LoginUserInfo) bool {
        return u.UserPermission == Admin || u.UserPermission == SuperAdmin
    }, DefaultForbidden)(ctx)
}

func RequireSuperAdmin(ctx *gin.Context) {
    RequirePermission(func(u *LoginUserInfo) bool {
        return u.UserPermission == SuperAdmin
    }, DefaultForbidden)(ctx)
}
```

### 3.3 上下文读取

中间件验证通过后，将用户信息写入 Gin Context：

| Key | 类型 | 读取方法 |
|---|---|---|
| `login_user` | `*LoginUserInfo` | `GetLoginUser(ctx)` |
| `login_uid` | `int64` | `GetLoginUID(ctx)` |
| `login_token` | `string` | `GetLoginToken(ctx)` |

Business 层通过 `GetLoginUIDFromContext(ctx)` 获取 UID，内部通过 `longgin.GetGinContext` 从 `context.Context` 中提取 Gin Context。

## 四、登录流程

```
客户端                     Business                    数据库/缓存
  │                          │                            │
  │  LoginRequest            │                            │
  │  (account, password,     │                            │
  │   device_type)           │                            │
  │─────────────────────────→│                            │
  │                          │  查用户（账号/手机/邮箱）    │
  │                          │───────────────────────────→│
  │                          │  校验密码                   │
  │                          │                            │
  │                          │  生成 JWT                   │
  │                          │  生成随机 Token             │
  │                          │                            │
  │                          │  存储 Token 记录            │
  │                          │───────────────────────────→│
  │                          │                            │
  │                          │  写缓存                     │
  │                          │───────────────────────────→│
  │                          │                            │
  │  LoginResponse           │                            │
  │  (token, user)           │                            │
  │←─────────────────────────│                            │
```

### 4.1 密码校验

```go
err = bcrypt.CompareHashAndPassword([]byte(u.Password), []byte(req.Password))
```

使用 bcrypt 哈希比对，不存储明文密码。

### 4.2 多设备登录

每次登录生成一条 `tb_user_token` 记录，同一用户可以有多个设备的 Token 同时有效。

踢下线时按 `device_type` 过滤删除。

## 五、注册流程

```
1. 校验参数（账号/密码/验证码）
2. 检查账号是否已存在
3. bcrypt 加密密码
4. 创建用户记录
5. 初始化用户设置
6. 自动登录（生成 Token）
```

## 六、踢下线流程

### 6.1 用户自己踢设备

```
1. 获取当前用户所有 Token 记录
2. 按 device_type 过滤
3. 删除数据库记录
4. 删除缓存
```

### 6.2 管理员踢人

```
1. 获取目标用户所有 Token 记录
2. 全部删除
3. 清空该 UID 的所有缓存
```

## 七、路由配置

```go
// 不需要登录的路由 — 不注册中间件
publicGroup := engine.Group("")

// 需要登录的路由 — 注册 Middleware
userGroup := engine.Group("", middlewares.Middleware)

// 需要管理员的路由 — 注册 Middleware + RequireAdmin
adminGroup := engine.Group("", middlewares.Middleware, middlewares.RequireAdmin)

// 需要超管的路由 — 注册 Middleware + RequireSuperAdmin
superAdminGroup := engine.Group("", middlewares.Middleware, middlewares.RequireSuperAdmin)
```

## 八、前端统一认证组件 (auth.js)

### 8.1 设计原则

核心原则：**前端统一带 Token，后端决定是否校验**。

前端不需要知道"这个接口要不要带 Token"，所有请求都带，后端不用就忽略。

### 8.2 常量定义

与后端一一对应：

| 前端常量 | 值 | 后端对应 |
|---|---|---|
| `PERMISSION.NORMAL` | 0 | `model/user.NORMAL` |
| `PERMISSION.ADMIN` | 1 | `model/user.ADMIN` |
| `PERMISSION.SUPER_ADMIN` | 2 | `model/user.SUPER_ADMIN` |
| `RESPONSE_CODE.UNAUTHORIZED` | -401 | `auth.CodeUnauthorized` |
| `RESPONSE_CODE.FORBIDDEN` | -403 | `auth.CodeForbidden` |

### 8.3 Token 管理

```javascript
Auth.setToken(token)       // 登录成功后保存 Token 到 localStorage
Auth.getToken()            // 读取 Token
Auth.clearToken()          // 登出/Token 失效时清除
Auth.isAuthenticated()     // 判断是否已登录
```

### 8.4 Auth.fetchApi() — 统一 API 调用

所有页面必须通过 `Auth.fetchApi()` 调用后端接口，不再各自实现 fetch：

```javascript
// 调用方式
const result = await Auth.fetchApi(baseUrl, action, params);

// 示例
const result = await Auth.fetchApi('/auth', 'login', { account, password });
const result = await Auth.fetchApi('/settings', 'setDesktopApp', { uid, app_id, x, y });
const result = await Auth.fetchApi('/long_store', 'uninstall', { app_id });
```

内部流程：

```
1. 拼接 URL: baseUrl + '/' + action
2. 自动从 localStorage 读取 Token，写入 Authorization Header
3. POST JSON 请求
4. 解析响应：
   - code === -401 → 清 Token + 跳首页
   - code === -403 → 弹窗 + 返回上页
   - code < 0      → throw Error
   - code >= 0     → 返回 result
```

### 8.5 权限判断

```javascript
Auth.getUserPermission()   // 从 JWT 解析 user_permission 字段
Auth.isSuperAdmin()        // permission >= 2
Auth.isAdmin()             // permission >= 1
Auth.requirePermission(level)  // 前端权限拦截，不足则弹窗+返回
```

权限值从 JWT Payload 的 `user_permission` 字段读取，无需额外请求。

### 8.6 页面接入方式

```html
<!-- 1. 引入 auth.js -->
<script src="/static/common/auth.js"></script>
<script src="main.js"></script>
```

```javascript
// 2. 调用 API — 自动带 Token，自动处理错误
const result = await Auth.fetchApi('/api_catcher', 'list', { limit: 50 });

// 3. 判断权限
if (Auth.isSuperAdmin()) {
    // 显示超管专属按钮
}
```

### 8.7 重构前后对比

| | 重构前 | 重构后 |
|---|---|---|
| API 调用 | 各页面自己 `fetch` + 手动拼 Header | `Auth.fetchApi()` 统一调用 |
| Token 管理 | 各页面自己 `localStorage` | `Auth.setToken()` / `Auth.clearToken()` |
| -401 处理 | 有的页面写 `code === 401`（少负号） | `Auth.fetchApi` 自动处理 |
| -403 处理 | 大部分页面不处理 | `Auth.fetchApi` 自动处理 |
| 权限判断 | 硬编码 `=== 1` / `=== 2` | `Auth.isSuperAdmin()` / `Auth.isAdmin()` |
| 新增页面 | 复制粘贴 fetch 逻辑 | 引入 auth.js 即可 |
