## 核心问题

### 1. common/auth 职责混乱

`common/auth` 名字暗示"通用"，但实现绑定了业务：

```go
// 直接嵌入应用模型
type LoginUserInfo struct {
    *user.TbUser           // ← 绑死了应用模型
    UserPermission int
}

// Claims 硬编码了应用字段
type Claims struct {
    Uid            int64  `json:"uid"`
    UserPermission int    `json:"user_permission"` // ← 不是所有项目都有
    ...
}

// 密钥硬编码
var jwtSecret = []byte("armylong-secret-key-2024") // ← 不该出现在代码里

// 中间件直接查数据库
tokenRecord, err := user.TbUserTokenModel.GetByToken(token)    // ← 查库
u, err := user.TbUserModel.GetByUid(tokenRecord.Uid)           // ← 查库
permission := user.TbAdminUserModel.GetUserPermission(u.Uid)    // ← 查库
```

**定位和实现不一致**：放在 common 下暗示通用，但实现全是应用特有的。

### 2. auth 和 user_management 人为割裂

`business/auth` 和 `business/user_management` 操作的都是同一个 `user` 聚合根：

- `user_management` 禁用用户时调用 `auth.Kickoff()`
- `user_management` 踢人时调用 `auth.Kickoff()`
- 改密码逻辑分散在 `auth.ChangePassword` 和 `user_management.UpdatePassword` 两处

同一个领域的操作被拆成两个包，导致互相依赖和逻辑重复。

### 3. 中间件越权

`common/auth` 的 Middleware 里直接操作数据库，基础设施层不应该知道"用户怎么查"、"权限怎么算"。

### 4. controller 层碎片化

三个 controller 包操作同一个领域：
- `controllers/auth` — 登录/注册/登出/踢下线
- `controllers/user` — 用户信息/改密码
- `controllers/user_management` — 管理员操作

路由注册时需要 import 三个包，维护成本高。

## 依赖关系

```
controllers/auth → business/auth → common/auth → model/user
controllers/user → business/auth → common/auth → model/user
                                  common/auth → model/user (中间件直接查库)
controllers/user_management → business/user_management → business/auth → common/auth
```

`common/auth` 是全局耦合点，所有层都依赖它，而它又反向依赖 `model/user`。
