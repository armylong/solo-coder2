# 用户认证 - 重构后设计

## 目录结构

```
go-library/service/auth/          ← 通用基础设施（可复用）
  ├── jwt.go                      JWT 生成/解析，secret 可配置
  ├── cache.go                    泛型 Token 缓存，sync.Map 实现
  └── middleware.go               中间件骨架，通过回调注入业务逻辑

armylong-go/internal/
├── middlewares/auth.go           ← 应用层中间件组装
│                                  LoginUserInfo 定义
│                                  Middleware / RequireAdmin / RequireSuperAdmin
│                                  GetLoginUID / GetLoginUser / GetLoginToken
│                                  SetCache / DeleteCache / ClearCacheByUID
│                                  UserLoader 实现
│
├── business/user/                ← 用户领域（合并后）
│   ├── types.go                  所有请求/响应类型
│   ├── login.go                  登录/登出
│   ├── register.go               注册
│   ├── change_password.go        修改密码
│   ├── kickoff.go                踢下线
│   └── management.go             用户管理 CRUD
│
├── controllers/user/             ← 用户控制器（合并后）
│   ├── user_controller.go        AuthController + UserController
│   │                             + UserManagementController
│   └── demo_controller.go        DemoController
│
├── cs/user/                      ← 用户请求/响应类型
│
├── model/user/                   ← 用户数据模型（不变）
│   ├── tb_user.go

│   ├── tb_admin_user.go
│   └── tb_user_settings.go
│
└── static/common/auth.js         ← 前端统一认证组件
                                   所有页面共用，不再各自实现
```

## 依赖关系

### 后端

```
controllers/user → business/user → middlewares/auth → go-library/service/auth
                                     ↑                    ↑
                                     └── model/user       └── 无业务依赖
```

### 前端

```
所有页面 → static/common/auth.js → 后端 API
              ↑
              唯一入口，不再各自实现 fetch/Token/权限判断
```

对比重构前：

```
重构前：common/auth 是全局耦合点，所有层依赖它，它又反向依赖 model/user
        前端各页面各自实现 fetchWithAuth / Token 管理 / 权限判断，逻辑分散且不一致
重构后：依赖方向清晰，library 无业务依赖，middlewares 是唯一的组装点
        前端统一走 auth.js，所有请求自动带 Token、自动处理 -401/-403
```

## 各层职责

### go-library/service/auth/ — 通用基础设施

只提供"能力"，不知道任何业务：

| 文件 | 提供的能力 |
|---|---|
| `jwt.go` | `GenerateToken(config, uid, name, deviceType, duration)` — JWT 生成 |
| | `ParseToken(config, tokenString)` — JWT 解析 |
| | `GenerateRandomToken()` — 随机 Token |
| `cache.go` | `TokenCache[T]` 接口 — 缓存抽象 |
| | `SyncMapCache[T]` — 默认实现 |
| `middleware.go` | `Middleware[T](config)` — 登录验证中间件 |
| | `RequirePermission[T](check, unauthorized)` — 权限检查中间件 |
| | `GetLoginUser[T](ctx)` / `GetLoginUID(ctx)` / `GetLoginToken(ctx)` — 上下文读取 |
| | `IsTokenExpired(expireAt)` — 过期判断 |
| | `DefaultUnauthorized` / `DefaultForbidden` — 默认响应 |
| | `CodeUnauthorized` (-401) / `CodeForbidden` (-403) — 响应码常量 |

### internal/middlewares/auth.go — 应用层组装

用 library 的骨架 + 应用层的回调，组装出具体可用的中间件：

```go
type LoginUserInfo struct {
    *user.TbUser
    UserPermission int
}

middlewareConfig = MiddlewareConfig[*LoginUserInfo]{
    JWTConfig:    jwtConfig,
    Cache:        loginUserInfoCache,
    UserLoader:   loadUserByToken,
    GetUID:       func(u) int64 { return u.Uid },
    IsZero:       func(u) bool { return u == nil },
    Unauthorized: DefaultUnauthorized,
}
```

### static/common/auth.js — 前端统一认证组件

所有页面共用的唯一认证入口，不再各自实现：

| 方法 | 功能 |
|---|---|
| `Auth.getToken()` | 读取 Token |
| `Auth.setToken(token)` | 保存 Token |
| `Auth.clearToken()` | 清除 Token |
| `Auth.isAuthenticated()` | 是否已登录 |
| `Auth.getUserPermission()` | 获取当前用户权限等级 |
| `Auth.isSuperAdmin()` | 是否超管 (≥2) |
| `Auth.isAdmin()` | 是否管理员 (≥1) |
| `Auth.requirePermission(level)` | 前端权限拦截 |
| `Auth.fetchWithAuth(url, options)` | 带 Token 的 fetch |
| `Auth.fetchApi(baseUrl, action, params)` | 统一 API 调用，自动处理 -401/-403 |

| 常量 | 值 | 说明 |
|---|---|---|
| `PERMISSION.NORMAL` | 0 | 普通用户 |
| `PERMISSION.ADMIN` | 1 | 管理员 |
| `PERMISSION.SUPER_ADMIN` | 2 | 超级管理员 |
| `RESPONSE_CODE.UNAUTHORIZED` | -401 | 未登录 |
| `RESPONSE_CODE.FORBIDDEN` | -403 | 无权限 |

## 前后端协作规则

### 核心原则：前端统一带 Token，后端决定是否校验

```
前端：所有请求都走 Auth.fetchApi()，自动带 Token
后端：需要登录的路由加中间件，不需要的不加
```

前端不需要知道"这个接口要不要带 Token"，带就完了，后端不用就忽略。

### 新增页面只需三步

1. HTML 里引入 `<script src="/static/common/auth.js"></script>`
2. 用 `Auth.fetchApi(baseUrl, action, params)` 调接口
3. 用 `Auth.isSuperAdmin()` / `Auth.isAdmin()` 判断权限

### 后端路由配置

```go
// 不需要登录的路由 — 不注册中间件
publicGroup := engine.Group("")

// 需要登录的路由 — 注册 Middleware
userGroup := engine.Group("", middlewares.Middleware)

// 需要管理员的路由 — 注册 Middleware + RequireAdmin
adminGroup := engine.Group("", middlewares.Middleware, middlewares.RequireAdmin)

// 需要超管的路由 — 注册 Middleware + RequireSuperAdmin
superAdminGroup := engine.Group("", middlewares.Middleware, middlewares.RequireSuperAdmin)
```

### 错误码对应关系

| 后端常量 | 值 | 前端常量 | 前端处理 |
|---|---|---|---|
| `CodeUnauthorized` | -401 | `RESPONSE_CODE.UNAUTHORIZED` | 清 Token + 跳首页 |
| `CodeForbidden` | -403 | `RESPONSE_CODE.FORBIDDEN` | 弹窗 + 返回上页 |
