# 用户认证 - 重构思路

## 核心原则

**通用层只定义流程和接口，业务细节由应用层填充。**

## 分层归位

| 层 | 应该做什么 | 不应该做什么 |
|---|---|---|
| library | 提供可复用的能力骨架 | 知道任何业务模型 |
| middlewares | 用 library 骨架组装应用特有中间件 | 包含业务逻辑 |
| business | 编排业务流程（用例） | 依赖 gin 框架 |
| controllers | HTTP 协议处理 | 包含业务逻辑 |

## 三个关键决策

### 决策一：common/auth 拆分

`common/auth` 混杂了两种东西：
- **通用能力**：JWT 生成/解析、Token 缓存、中间件流程 → 下沉到 `go-library/service/auth/`
- **业务实现**：LoginUserInfo 类型、查库加载用户、权限判断 → 上浮到 `middlewares/` 和 `business/user/`

拆分方式：

```
common/auth (278行)
    │
    ├── JWT 生成/解析 ──────→ go-library/service/auth/jwt.go
    │                         secret 可配置，Claims 通用化
    │
    ├── Token 缓存 ─────────→ go-library/service/auth/cache.go
    │                         泛型 TokenCache[T]，不绑定模型
    │
    ├── 中间件骨架 ──────────→ go-library/service/auth/middleware.go
    │                         通过 UserLoader/GetUID/IsZero 回调注入
    │
    ├── RequireAdmin 等 ────→ go-library/service/auth/middleware.go
    │                         RequirePermission[T] 通用权限检查
    │
    ├── LoginUserInfo ──────→ internal/middlewares/auth.go
    │                         应用层定义自己的用户类型
    │
    └── 查库逻辑 ───────────→ internal/middlewares/auth.go
                              UserLoader 回调实现
```

### 决策二：auth + user_management 合并为 user 领域

auth 不是一个独立的业务领域，它是 user 领域的一个切面。

登录、注册、改密码、踢下线、用户管理——这些操作的都是同一个 `user` 聚合根。强行拆成两个包就像把"付款"从"订单"里拆出来，看似清晰，实则割裂。

合并方式：

```
business/auth/              ─┐
  login.go                   │
  register.go                │
  change_password.go         ├──→ business/user/
  kickoff.go                 │      login.go
  types.go                   │      register.go
                             │      change_password.go
business/user_management/  ──┤      kickoff.go
  user_management.go         │      management.go
                             │      types.go
```

### 决策三：controller 合并

三个 controller 包操作同一个领域，合并为一个：

```
controllers/auth/           ─┐
  auth_controller.go         │
                             ├──→ controllers/user/
controllers/user/           ──┤      user_controller.go
  user_controller.go         │      (含 AuthController + UserController
                             │       + UserManagementController)
controllers/user_management/─┘      demo_controller.go
  user_management_controller.go
```

## library 层的泛型设计

为了让 `service/auth` 真正通用，使用泛型消除对业务模型的依赖：

```go
// 缓存接口 - T 可以是任何用户类型
type TokenCache[T any] interface {
    Set(token string, user T)
    Get(token string) T
    Delete(token string)
    DeleteByTokens(tokens []string)
    ClearByUID(uid int64, getUID func(T) int64)
}

// 中间件配置 - 通过回调注入业务逻辑
type MiddlewareConfig[T any] struct {
    JWTConfig    JWTConfig
    Cache        TokenCache[T]
    UserLoader   func(token string) (T, error)   // 怎么加载用户
    GetUID       func(T) int64                    // 怎么取 UID
    IsZero       func(T) bool                     // 怎么判断空值
    Unauthorized func(ctx *gin.Context, msg string)
}
```

这样其他项目接入时，只需定义自己的用户类型，实现几个回调函数即可。
