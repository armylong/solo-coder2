# 龙聊 - 数据模型

## 设计理念

单聊和群聊统一为 chat 实体，通过 chat_type 区分。单聊本质是 2 人群聊，可无缝升级为群聊。

参考：Discord 的 channel 设计，所有聊天都是 channel，私聊是 type=DM 的 channel。

## 表结构

### tb_long_chat — 聊天实体

| 字段 | 类型 | 说明 |
|---|---|---|
| chat_id | TEXT PRIMARY KEY | 雪花ID |
| chat_type | INTEGER DEFAULT 1 | 1-单聊, 2-群聊 |
| chat_name | TEXT | 群聊存群名, 单聊可为空 |
| owner_uid | INTEGER | 群主uid, 单聊时为发起人uid |
| status | INTEGER DEFAULT 1 | 1-正常, 2-已解散 |
| created_at | DATETIME | |
| updated_at | DATETIME | |

### tb_long_chat_member — 聊天成员

只存当前成员，离开即删除。加群/退群历史通过系统消息追溯。

| 字段 | 类型 | 说明 |
|---|---|---|
| chat_id | TEXT | 联合主键 |
| uid | INTEGER | 联合主键 |
| joined_at | DATETIME | |

PRIMARY KEY (chat_id, uid)

### tb_long_chat_list — 用户聊天列表

每个用户每个会话一条记录，用于渲染聊天列表。冗余 chat_name 避免查询时 JOIN。

| 字段 | 类型 | 说明 |
|---|---|---|
| chat_id | TEXT | 联合主键 |
| uid | INTEGER | 联合主键 |
| chat_name | TEXT | 单聊存对方昵称, 群聊存群名 |
| unread | INTEGER DEFAULT 0 | 未读数 |
| last_msg | TEXT | 最新消息摘要 |
| last_msg_at | DATETIME | 最新消息时间 |
| created_at | DATETIME | |

PRIMARY KEY (chat_id, uid)

### tb_long_chat_message — 消息

| 字段 | 类型 | 说明 |
|---|---|---|
| msg_id | TEXT PRIMARY KEY | 雪花ID |
| chat_id | TEXT | 关联会话 |
| from_uid | INTEGER | 发送者uid, 系统消息时为0 |
| msg_type | TEXT DEFAULT 'text' | text/image/voice/system/... |
| content | TEXT | 文本时为纯文本, 非文本时为JSON |
| created_at | DATETIME | |

### tb_long_chat_friend — 好友关系

| 字段 | 类型 | 说明 |
|---|---|---|
| uid | INTEGER | 联合主键 |
| friend_uid | INTEGER | 联合主键 |
| status | INTEGER DEFAULT 1 | 1-待确认, 2-已通过, 3-已拒绝 |
| created_at | DATETIME | |

PRIMARY KEY (uid, friend_uid)

## 关键设计决策

### 为什么单聊群聊统一？

- 单聊升群聊无需迁移数据，直接加人即可
- 成员统一走 member 表，无分支逻辑
- Discord 同款设计，经过大规模验证

### 为什么 list 表冗余 chat_name？

聊天列表是最高频操作，冗余后一条 SQL 搞定：

```sql
SELECT * FROM tb_long_chat_list WHERE uid = ? ORDER BY last_msg_at DESC
```

代价：用户改名时需批量更新相关单聊的 chat_name。改名是低频操作，可接受。

### 为什么 member 表直接删而不加状态？

退群/被踢就是离开了，没有中间状态。加状态后所有查询都要 WHERE status = 1，变复杂。加群/退群提示通过系统消息实现，不依赖 member 表。

### 为什么消息表没有 to_uid？

群聊的接收者是所有成员，从 member 表查，消息表不需要存。单聊同理，从 member 表查对方。

### msg_type 与 content 的关系

| msg_type | content 格式 | 示例 |
|---|---|---|
| text | 纯文本 | 你好 |
| image | JSON | {"url":"https://xxx/a.jpg","width":800,"height":600} |
| voice | JSON | {"url":"https://xxx/voice.mp3","duration":5} |
| system | 纯文本 | 张三加入了群聊 |

文本不包 JSON，非文本包 JSON。msg_type 做独立字段便于建索引和按类型筛选。

## ID 生成

所有 ID（chat_id、msg_id）使用雪花算法生成，保证全局唯一，支持未来分布式部署。

机器 ID 基于 IP 地址计算（IP 转整数 % 1024），无需手动配置。
